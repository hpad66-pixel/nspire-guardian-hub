Step 1 — Prime Lovable once. Before any module prompt, paste the Context Primer below into Lovable as a single message. It establishes the stack, conventions, naming rules, and the tenant-id pattern so every subsequent prompt inherits them.

Step 2 — Run prompts in order. The 29 prompts are grouped by build phase. Do not skip ahead — Phase 2 prompts assume Phase 1 tables exist, Phase 3 prompts assume Cost Codes exist, and so on. The DEPENDS ON line at the top of each prompt is the gate.

Step 3 — One prompt per Lovable turn. Paste a full prompt, wait for Lovable to scaffold files + migrations, review the diff, run tests, then commit. Move to the next.

Step 4 — Seed data after each Phase. At the end of each phase there's a Seed prompt that inserts demo data so you can QA the UI in a live tenant.

  • TABLES — the Postgres DDL Lovable should add as a Supabase migration, including RLS.
  • COMPONENTS — file paths + purposes. Match existing src/components/... folder conventions.
  • ROUTES — React Router paths to register in src/App.tsx.
  • BUSINESS RULES — trigger functions, workflow assertions, derived fields.
  • ACCEPTANCE TESTS — what must pass before moving on.

Multi-Tenant Isolation Hardening
Phase 1
Depends on: existing tenants/workspaces tables. Run first in Phase 1 — every other prompt assumes this is enforced.
GOAL: Harden multi-tenant isolation so no tenant can ever read another tenant's data.

TABLES:

-- Audit existing tables; for any missing tenant_id, add with backfill.

-- 2. Tenant settings (per-tenant feature flags, defaults)

  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id uuid NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
  key text NOT NULL,
  value jsonb NOT NULL,
  updated_at timestamptz DEFAULT now(),
  UNIQUE(tenant_id, key)
);
ALTER TABLE tenant_settings ENABLE ROW LEVEL SECURITY;
CREATE POLICY ts_read ON tenant_settings FOR SELECT
  USING (tenant_id = (auth.jwt() ->> 'tenant_id')::uuid);
CREATE POLICY ts_write ON tenant_settings FOR ALL
  USING (tenant_id = (auth.jwt() ->> 'tenant_id')::uuid)
  WITH CHECK (tenant_id = (auth.jwt() ->> 'tenant_id')::uuid);

Spec A4 · Phase 1
Ball-in-Court Workflow Engine
Phase 1
Depends on: A1, A2. Blocks: every workflow module (RFI, Submittal, Punch, CO, Meeting).
GOAL: Generic workflow engine. Every workflow-bearing module creates a
workflow_instances row. User dashboard shows "My Court" across all modules.

TABLES:
CREATE TABLE workflow_definitions (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id uuid NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
  module text NOT NULL,                 -- 'rfi' | 'submittal' | ...
  name text NOT NULL,
  version int NOT NULL DEFAULT 1,
  is_default boolean DEFAULT false,
  UNIQUE(tenant_id, module, version)
);
CREATE TABLE workflow_steps (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  definition_id uuid NOT NULL REFERENCES workflow_definitions(id) ON DELETE CASCADE,
  sequence int NOT NULL,
  state_name text NOT NULL,
  assignee_rule text NOT NULL,           -- 'role:PM' | 'field:responsible_contractor_id' | 'explicit'
  due_offset_days int,
  auto_actions jsonb DEFAULT '{}'::jsonb,
  UNIQUE(definition_id, sequence)
);
CREATE TABLE workflow_instances (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id uuid NOT NULL,
  definition_id uuid NOT NULL REFERENCES workflow_definitions(id),
  record_id uuid NOT NULL,
  record_type text NOT NULL,
  project_id uuid REFERENCES projects(id),
  current_step int NOT NULL,
  current_assignee_id uuid REFERENCES auth.users(id),
  due_at timestamptz,
  state text NOT NULL,                   -- 'open' | 'approved' | 'closed' | 'rejected'
  created_at timestamptz DEFAULT now(),
  closed_at timestamptz,
  INDEX idx_court (current_assignee_id, state) WHERE state = 'open'
);
CREATE TABLE workflow_events (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  instance_id uuid NOT NULL REFERENCES workflow_instances(id) ON DELETE CASCADE,
  from_step int,
  to_step int,
  actor_id uuid REFERENCES auth.users(id),
  action text NOT NULL,                  -- 'submit' | 'approve' | 'reject' | 'return' | 'close'
  comment text,
  occurred_at timestamptz DEFAULT now()
);
-- + RLS tenant_id

COMPONENTS:
- src/pages/dashboard/MyCourtPage.tsx — all open instances assigned to me
- src/components/workflow/BallInCourtBadge.tsx — shows current assignee + due
- src/components/workflow/WorkflowHistoryTimeline.tsx — events chronology
- src/components/admin/WorkflowEditor.tsx — Owner customizes steps per module
- src/hooks/useWorkflow.ts — create instance, advance, my-court query
- src/lib/workflow-engine.ts — state transition functions

ROUTES:
- /dashboard/my-court
- /admin/workflows

BUSINESS RULES:
- Seed default definitions for: rfi, submittal, punch_item, change_order,
  potential_change_order, prime_change_order, commitment_change_order,
  prime_pay_application, commitment_pay_application, inspection
- When a workflow-bearing record is created, call
  `create_workflow_instance(record_id, record_type, definition_id)`
- On user action, call `advance_workflow(instance_id, action, comment)` which
  writes event row and updates current_step + current_assignee_id + due_at
- Nightly job: escalate overdue instances (configurable per tenant)

ACCEPTANCE TESTS:
- Create RFI — workflow_instance created with current_step=1 assigned to RFI Manager
- Dashboard "My Court" shows the RFI for the assigned user

-- The matrix view
CREATE OR REPLACE VIEW budget_matrix AS
SELECT bl.project_budget_id,
       bl.cost_code_id,
       cc.code AS cost_code,
       cc.description AS cost_code_desc,
       bl.original_budget,
       COALESCE(bm_net.net_transfer, 0) AS approved_budget_mods,
       bl.original_budget + COALESCE(bm_net.net_transfer, 0) AS revised_budget,
       COALESCE(cs.committed_cost, 0) AS committed_cost,
       COALESCE(cs.executed_cco, 0) AS executed_cco,
       COALESCE(dc.direct_cost, 0) AS direct_cost,
       COALESCE(cev.pending_exposure, 0) AS pending_exposure,
       (COALESCE(cs.committed_cost,0) + COALESCE(cs.executed_cco,0)
        + COALESCE(dc.direct_cost,0) + COALESCE(cev.pending_exposure,0)) AS forecast_to_complete,
       (bl.original_budget + COALESCE(bm_net.net_transfer, 0))
        - (COALESCE(cs.committed_cost,0) + COALESCE(cs.executed_cco,0)
           + COALESCE(dc.direct_cost,0) + COALESCE(cev.pending_exposure,0)) AS variance
FROM budget_lines bl
JOIN cost_codes cc ON cc.id = bl.cost_code_id
LEFT JOIN LATERAL (
  SELECT
    SUM(CASE WHEN bml.to_cost_code_id=bl.cost_code_id   THEN bml.amount ELSE 0 END) -
    SUM(CASE WHEN bml.from_cost_code_id=bl.cost_code_id THEN bml.amount ELSE 0 END) AS net_transfer
  FROM budget_modification_lines bml
  JOIN budget_modifications bm ON bm.id=bml.budget_modification_id
  WHERE bm.status='approved'
) bm_net ON true
LEFT JOIN LATERAL (
  SELECT
    SUM(csl.scheduled_value) AS committed_cost,
    SUM(col.amount) FILTER (WHERE co.status='executed' AND co.co_type='CCO') AS executed_cco
  FROM commitment_sov_lines csl
  LEFT JOIN commitments c ON c.id=csl.commitment_id AND c.status='executed'
  LEFT JOIN change_order_lines col ON col.cost_code_id=bl.cost_code_id
  LEFT JOIN change_orders co ON co.id=col.change_order_id
  WHERE csl.cost_code_id=bl.cost_code_id
) cs ON true
LEFT JOIN LATERAL (
  SELECT SUM(dcl.amount) AS direct_cost
  FROM direct_cost_lines dcl
  JOIN direct_costs dc ON dc.id=dcl.direct_cost_id AND dc.status IN ('approved','paid')
  WHERE dcl.cost_code_id=bl.cost_code_id
) dc ON true
LEFT JOIN LATERAL (
  SELECT SUM(cel.estimated_cost) AS pending_exposure
  FROM change_event_lines cel
  WHERE cel.cost_code_id=bl.cost_code_id AND cel.status_bucket='pending'
) cev ON true;
